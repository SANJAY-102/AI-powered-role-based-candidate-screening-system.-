1. Objective

The goal of this project was to build a fully functional, end-to-end technical interview simulation platform. The system is designed to screen candidates for specific engineering roles by generating adaptive, domain-grounded questions using a Retrieval-Augmented Generation (RAG) pipeline, and scoring candidate responses in real time.

The system needed to go beyond a simple chatbot. It had to parse a candidate's resume, understand their experience level and technical background, retrieve relevant knowledge from a curated knowledge base, and then ask intelligent follow-up questions that match the candidate's profile.

2. Problem Statement

Traditional candidate screening is time-consuming, inconsistent, and heavily dependent on the availability of senior engineers. There is a clear need for an automated system that can conduct a structured technical interview, adjust question difficulty based on candidate profile, evaluate responses objectively, and generate a final performance report without any human intervention.

The specific challenges this system addresses are:

  • How to generate role-specific questions that are grounded in real technical knowledge, not hallucinated by a language model.
  • How to personalize difficulty based on resume data.
  • How to score open-ended answers fairly and consistently.
  • How to do all of the above without requiring the candidate to depend on a single API provider.
3. System Architecture

The system is divided into two main layers: a Python backend powered by FastAPI, and a React frontend. These communicate via a REST API. The backend orchestrates all logic — resume parsing, vector retrieval, question generation, and scoring.

3.1 RAG Pipeline

The RAG pipeline is the core of this system. It ensures questions are technically accurate and grounded in domain knowledge rather than being fabricated by the language model. The pipeline runs in five stages:

  1. Ingest: PyMuPDF extracts raw text from PDF documents in the knowledge base directory.
  2. Chunk: The extracted text is split into overlapping segments of approximately 400 tokens each, preserving context boundaries.
  3. Embed: Each chunk is encoded into a 384-dimensional dense vector using the all-MiniLM-L6-v2 model from sentence-transformers.
  4. Index: All vectors are stored in a FAISS flat index for fast approximate nearest-neighbour retrieval.
  5. Generate: At interview time, a query is built from the candidate's role and skills, the top-k most similar chunks are retrieved, and these are passed as context to GPT-3.5-turbo to generate a targeted question.
3.2 Scoring

When a candidate submits an answer, it is sent alongside the original question and the retrieved knowledge chunk to GPT-3.5-turbo. The model evaluates the answer on a scale of 1 to 5 and returns a brief explanation. If the OpenAI API is unavailable, a local keyword-overlap heuristic is used as fallback.

5. Features
5.1 Resume Parsing

Candidates upload a PDF resume. The backend extracts raw text using PyMuPDF, then runs keyword matching and pattern detection to identify technical skills, frameworks, programming languages, and approximate experience level (junior, mid, senior). This data seeds the RAG query and calibrates question difficulty.

If a candidate does not have a resume available, the system provides a mock profile option that generates a realistic sample profile for the selected role.

5.2 Role-Based Adaptive Questioning

Three engineering roles are supported: AI/ML Engineer, Backend Engineer, and Data Scientist. Questions are not fixed or templated — they are generated fresh each time using the RAG pipeline, so no two sessions produce identical questions.

5.3 Interviewer Personas

Three interviewer personas are available: Professional Mentor (structured and encouraging), Stress Interviewer (high-pressure, critical), and Academic Professor (theory-heavy and rigorous). The persona is injected into the GPT system prompt to shape the tone and style of questions.

5.4 Real-Time Scoring and Feedback

After each answer submission, a score from 1 to 5 and a short explanation is returned within seconds. The score is displayed as a notification and stored in the session database for inclusion in the final report.

5.5 Voice Features

The interview interface includes Text-to-Speech (TTS) — questions are read aloud using the browser's Web Speech API — and Speech-to-Text (STT), allowing candidates to dictate their answers verbally. Both features can be toggled individually in the settings panel.

5.6 Session Recovery

Every interview session is assigned a UUID. If a session is interrupted, the candidate can re-enter the session ID to resume exactly where they left off, including the question index and previously submitted answers.

5.7 Evaluation Report

Once all questions are answered, a detailed performance report is generated. It includes an overall score displayed as an SVG gauge, per-question score breakdown with expandable answer transcripts, topic coverage tags, and the option to export the full transcript as a PDF.

8. Setup and Installation
8.1 Prerequisites
  • Python 3.10 or higher
  • Node.js 18 or higher
  • OpenAI API key (optional — the system works without one via local fallbacks)
8.2 Backend Setup
cd backend
pip install -r requirements.txt

# Create .env file
# OPENAI_API_KEY=sk-...  (optional)

# Build the FAISS knowledge index
python create_sample_kb.py
python rag/embedder.py

# Start the API server
python -m uvicorn main:app --reload --port 8000
8.3 Frontend Setup
cd frontend
npm install
npm run dev

Open http://localhost:5173 in a browser. The backend must be running on port 8000.

9. Design Decisions

SQLite instead of PostgreSQL. For a single-machine deployment of this scope, SQLite removes infrastructure overhead. The ORM layer abstracts the database engine, so switching to Postgres in production requires only a connection string change.

FAISS on CPU. FAISS does not require a GPU or any cloud service. The index is built once and queried locally, making the system fully self-contained. For a knowledge base of a few hundred chunks, CPU retrieval completes in under a millisecond.

all-MiniLM-L6-v2. At 80 MB, this model loads quickly and produces high-quality 384-dimensional embeddings suited for FAISS. It runs entirely on CPU and has strong semantic accuracy benchmarks relative to its size.

Offline fallbacks. The system works without an API key. Template-based questions are generated from role-specific topic lists, and answers are scored using keyword frequency against retrieved chunks. This ensures the system is always demonstrable.

Session UUID for recovery. Network interruptions or accidental tab closures are realistic during a long interview. Storing all state server-side with a recoverable ID means no progress is lost and the session can be resumed at any time.

10. Known Limitations
  • The knowledge base quality directly affects question quality. Sparse or low-quality PDFs lead to generic questions.
  • Resume parsing uses keyword matching and is not a trained NLP model. Non-standard resume formats may result in fewer skills being detected.
  • GPT-3.5-turbo scoring can occasionally be inconsistent for borderline answers. The local fallback is less accurate but always functional.
  • SQLite is not suitable for concurrent multi-user production deployments. A production build should use PostgreSQL with connection pooling.
  • The current implementation does not support multi-language resumes or non-English sessions.
11. Future Scope
  • Add user authentication and a recruiter dashboard to track candidates across sessions.
  • Integrate a coding challenge module with in-browser execution for software engineering roles.
  • Train a custom scoring model on human-labelled interview responses to reduce GPT dependency.
  • Support multi-round interview flows (HR screening, technical round, system design round).
  • Add proctoring features such as tab-switch detection and session time limits.
  • Export reports as formatted PDFs using a headless rendering library like Puppeteer or WeasyPrint.
